How DNS Works

A step-by-step explanation of how your computer finds the correct IP address for any website.

What Happens When You Type a Website Into Your Browser?

When you enter a domain name like google.com into your browser, your device begins a DNS lookup. This process finds the correct IP address so your browser knows where to send the request. DNS works behind the scenes in milliseconds, but several important steps happen along the way.

DNS works so quickly that most users never notice it — but without it, the internet would be nearly impossible to use.

Recursive vs. Iterative DNS Queries

DNS lookups can happen in two different ways: recursive and iterative. Most users never notice the difference, but understanding these two methods explains why DNS is so efficient.

  • Recursive query: Your resolver does all the work for you. It contacts each DNS server in the chain until it finds the answer.
  • Iterative query: Each DNS server responds with the “next place to look,” and the resolver continues the search.

Most modern DNS systems use a combination of both, depending on the server and the type of request.

Step-by-Step: How a DNS Lookup Works

  1. Your browser checks its cache. If it has recently visited the site, it may already know the IP address.
  2. Your operating system checks its cache. If the browser doesn’t know, your computer might.
  3. The recursive resolver is contacted. This is usually run by your ISP or a public DNS provider like Google or Cloudflare.
  4. The resolver checks its own cache. If it has the answer, the lookup ends here.
  5. The resolver asks a root server. The root server directs the resolver to the correct top-level domain (TLD) server.
  6. The resolver asks the TLD server. For example, the .com TLD server points to the authoritative server for google.com.
  7. The resolver asks the authoritative server. This server holds the official DNS records and returns the IP address.
  8. The resolver sends the IP address back to your device.
  9. Your browser connects to the website.

Even though this process involves multiple servers, it usually completes in under a hundred milliseconds thanks to caching and optimized routing.

DNS Caching: Why Lookups Are Usually Instant

DNS caching is one of the biggest reasons the internet feels fast. Instead of repeating the entire lookup process every time, DNS stores results temporarily at multiple layers:

  • Browser cache: Stores recently visited domains for a short time.
  • Operating system cache: Shared by all applications on your device.
  • Recursive resolver cache: Stores millions of DNS records for thousands of users.

Each cached record has a TTL (Time To Live) value that determines how long it stays valid. When the TTL expires, the resolver must perform a fresh lookup.

Common DNS Record Types

DNS uses different types of records to store information about a domain. Each record serves a specific purpose:

  • A Record: Maps a domain to an IPv4 address.
  • AAAA Record: Maps a domain to an IPv6 address.
  • CNAME Record: Points one domain to another domain.
  • MX Record: Specifies mail servers for a domain.
  • NS Record: Identifies the authoritative name servers.
  • TXT Record: Stores text data, often for security or verification.

These records work together to define how a domain behaves on the internet.

Why DNS Is So Fast

DNS is designed for speed. Most DNS lookups take only a few milliseconds because recursive resolvers maintain large caches and use optimized routing. Many DNS providers also use Anycast networks, which route your request to the closest available server.

  • Caching: Saves previous lookups to avoid repeating work.
  • Anycast routing: Sends your request to the nearest DNS server.
  • Distributed infrastructure: DNS servers exist all over the world.
  • Lightweight protocol: DNS messages are small and efficient.

This combination ensures that DNS rarely becomes a bottleneck, even during high traffic periods.

Example: DNS Lookup for google.com

To see DNS in action, here’s what happens when you visit google.com:

  1. Your browser checks its cache for google.com.
  2. Your operating system checks its DNS cache.
  3. Your recursive resolver checks its cache.
  4. If not found, the resolver asks a root server.
  5. The root server points to the .com TLD server.
  6. The TLD server points to Google’s authoritative DNS servers.
  7. The authoritative server returns multiple IP addresses for load balancing.
  8. Your browser connects to the fastest available Google server.

This entire process happens in a fraction of a second, thanks to caching and global DNS infrastructure.
